# Picocalc_NESco 音声出力ゲイン再設計

日付: 2026-04-22  
対象プロジェクト: `Picocalc_NESco`

## 目的

PicoCalc にはアナログボリュームがあるため、
ソフト側では不必要に音量を絞らず、
**音が破綻しない範囲でできるだけ大きく出す**
方針を採る。

この設計では、

- チャンネルごとの音色バランス変更は最小化する
- `Picocalc_InfoNES` 由来で実績のある `noise*4` は当面固定する
- 全体音量の調整点は 1 箇所に限定する

ことを目標とする。

## 現状の事実

- `platform/audio.c` の `InfoNES_SoundOutput()` で、
  `pulse1 + pulse2 + triangle + noise*4 + dpcm`
  の線形ミックスを行っている
- 同じ箇所で、
  `((mixed * 255u) + 640u) / 1280u`
  により `0..255` の 8bit 値へ正規化している
- その後、
  DC オフセット補正をかけて ring buffer に積む
- `drivers/pwm_audio.c` 側の最終出力は 8bit PWM、中心値は `128`
- `noise*4` は `Picocalc_InfoNES` 由来の既存実績があるため、
  当面の変更対象から外す
- PicoCalc 側にはアナログボリュームがあるため、
  ソフト側で過剰に安全側へ絞る必要は薄い

## 方針

### 1. 当面の調整対象は「全体ゲイン」だけに限定する

最初に触るのは
`platform/audio.c`
の正規化係数だけとする。

対象式:

```c
mix = ((mixed * 255u) + 640u) / 1280u;
```

ここで、

- `1280` を小さくする
- 丸め項 `640` は `divisor / 2` として一緒に調整する

ことで、
チャンネル比を変えずに全体音量を上げる。

### 2. `noise*4` は固定する

少なくとも第一段階では、

```c
noise * 4
```

は変更しない。

理由:

- 既存実績がある
- ここを触ると「音量」ではなく「音色バランス」の変更になる
- まずは全体ゲインだけを独立に評価したい

### 3. PWM 出力直前の増幅は行わない

8bit 化した後でゲインをかけると、
クリップしやすく音質面で不利なため、
PWM 出力直前では調整しない。

### 4. 将来の音量調整機能導入とは分ける

ユーザー向け音量設定は将来課題とし、
今回の目的は
**固定ミックスのデフォルト音量最適化**
に限定する。

## 実装段階

### フェーズ A: 正規化係数を定数化する

`platform/audio.c`
の正規化係数を、
式へ直接書かず定数名に切り出す。

例:

- `AUDIO_MIX_DIVISOR`
- `AUDIO_MIX_ROUND_BIAS`

これにより、
何を調整しているかをコード上で明確にする。

### フェーズ B: 全体ゲインを少しずつ上げる

最初の候補は、
現行 `1280` より少し小さい値を使う。

候補例:

- `1152`
- `1024`

ただし一度に大きくは下げず、
段階的に比較する。

### フェーズ C: ログ指標と聴感を分けて確認する

実機確認では、
`[AUDIO]`
と
`[AUDIO_MIX]`
の役割を分けて扱う。

`[AUDIO]`
で主に見るもの:

- `out_peak`
- `out_rms`
- `underruns`

`[AUDIO]`
で補助的に見るもの:

- `clipped`

`[AUDIO_MIX]`
で補助的に見るもの:

- `noise_peak`
- `dpcm_peak`
- `overrun`

聴感で確認するもの:

- BGM の聞こえやすさ
- 効果音の歪み
- 無音時ノイズ

`mix_peak`
は DC 中心化後の絶対値上限であり、
全体ゲイン比較の主指標には使わない。

## 実機確認の基準

### 代表確認 ROM

- `LodeRunner.nes`
  - 効果音と BGM の両方を確認しやすい
- `DragonQuest.nes`
  - RPG 系 BGM の聞こえやすさを見る
- `Xevious.nes`
  - 効果音と混在時のバランスを見る

### 成功条件

- 現状より明らかに小さすぎる印象が減る
- `[AUDIO]`
  の
  `out_peak`
  または
  `out_rms`
  が改善する
- 聴感上の歪みが目立たない
- 無音時ノイズが悪化しない
- `underruns`
  や
  `overrun`
  が悪化しない

補足:

- `1152`
  や
  `1024`
  のような候補帯では、
  `clipped`
  は大きく増えない可能性が高い
- したがって
  `clipped`
  は
  「危険信号の補助指標」
  として使い、
  成功判定の主軸にはしない

### 不採用条件

- 効果音や BGM で歪みが目立つ
- `[AUDIO]`
  の
  `clipped`
  が明確に増え始める
- `out_peak`
  や
  `out_rms`
  の伸びに対して聴感が悪化する
- 無音や小音量時の安定性が悪化する
- `underruns`
  または
  `[AUDIO_MIX] overrun`
  が悪化する

## 非目標

今回やらないこと:

- `noise*4` の見直し
- NES 非線形ミックスへの置換
- ユーザー設定の音量 UI
- PWM 出力直前の増幅

## 結論

今回の音量調整は、
**チャンネル比を変えずに全体ゲインだけを上げる**
方向で進める。

最初に触る場所は
`platform/audio.c`
の正規化係数であり、
`noise*4` は当面固定する。

実機評価では、
`[AUDIO]`
の
`out_peak`
/
`out_rms`
/
`underruns`
を主指標として見て、
`clipped`
と
`[AUDIO_MIX]`
の
`noise_peak`
/
`dpcm_peak`
/
`overrun`
は補助指標として扱う。
