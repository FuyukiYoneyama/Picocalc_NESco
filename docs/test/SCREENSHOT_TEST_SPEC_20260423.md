# スクリーンショット機能 テスト仕様書 2026-04-23

## 目的

スクリーンショット機能について、
何を確認できれば初期版完成とみなせるか、
正常系、
整合性、
異常系、
性能観点で固定する。

この文書では、
コード内容、
既存設計メモ、
ユーザー指示として確認できたものだけを
事実として書く。
そこから先の期待挙動や評価基準は
`【推定】`
を付ける。

## 入力前提

事実:
- ユーザー提示メモでは、
  スクリーンショット中のゲーム一時停止は許容されている。
- 同メモでは、
  capture 条件として
  `VBLANK`
  と
  LCD DMA 完了待ちを組み合わせる方針が示されている。
- 同メモでは、
  初期版の保存対象として
  `320x320`
  全画面 capture が推奨されている。
- 同メモでは、
  保存形式として
  `BMP`
  が有力とされている。
- ユーザー指示で、
  スクリーンショット実行キーは
  `F5`
  とすることが決定した。
- ユーザー指示で、
  `F5`
  の reset 割り当ては外し、
  `F1`
  を reset のまま残すことが決定した。

## 完了判定の基本

初期版では、
次を満たしたときに
「機能が成立した」
と判定する。

- game 中の要求で 1 枚保存できる
- capture 後に game が再開する
- `F5`
  で screenshot が起動し、
  reset が走らない
- `F1`
  は reset のまま維持される
- 生成された一般的なビューアで開ける
  `320x320`
  `24bpp BMP`
  が PC で開ける
- DMA 転送途中の破綻が目視で混ざらない

## 正常系テスト

### T01 単発保存

目的:
- 基本動作確認

手順:
1. game を起動する
2. `F5` を 1 回押す
3. 保存完了を待つ

期待結果:
- 1 ファイルだけ保存される
- `F5`
  入力で reset は走らない
- game が再開する
- busy 状態が解除される

### T02 次の VBLANK で capture

目的:
- capture 開始条件の確認

手順:
1. screenshot 最小ログを有効化する
2. screenshot 要求を入れる
3. capture 開始タイミングを記録する

期待結果:
- 要求直後ではなく、
  次の VBLANK 起点で capture が始まる
- ログ順が
  `SS_REQ -> SS_VBLANK -> SS_DMA_WAIT_BEGIN -> SS_CAPTURE_BEGIN`
  になる

### T03 連続 2 回保存

目的:
- 保存後の再実行確認

手順:
1. 1 枚保存する
2. 再開後にもう 1 回保存する

期待結果:
- 2 枚とも保存できる
- ファイル名衝突を起こさない

### T04 normal / stretch 両モード保存

目的:
- 表示モード差分の確認

手順:
1. normal 表示で保存する
2. stretch 表示へ切り替える
3. 再度保存する

期待結果:
- どちらも保存できる
- どちらも `320x320`
  BMP として開ける

## 整合性確認

### T11 DMA 転送途中の破綻が混ざらない

目的:
- DMA 完了待ちの有効性確認

観点:
- 画像の一部だけ古いフレームにならないこと
- strip 境界で段差や崩れが出ないこと
- ログ順が
  `SS_VBLANK -> SS_DMA_WAIT_BEGIN -> SS_DMA_WAIT_DONE -> SS_CAPTURE_BEGIN`
  になること

### T12 capture 中に画面が進行しない

目的:
- pause 設計の確認

観点:
- 保存中に game 状態が進まないこと
- 再開後に連続性が不自然に壊れないこと

### T13 UI hint / 黒帯込み確認

目的:
- 全画面保存方針の確認

観点:
- normal 表示で下部 hint が画像に含まれること
- 黒帯や余白が期待どおり保存されること

### T14 画像破損確認

目的:
- BMP のフォーマット妥当性確認

手順:
1. 生成ファイルを PC で開く
2. 別ビューアでも開く

期待結果:
- 一般的なビューアで開ける
- サイズ認識が `320x320`
  になっている

## 異常系テスト

### T21 SD 未挿入

目的:
- 保存先不在時の復帰確認

期待結果:
- ハングしない
- game は再開する
- busy が解除される

### T22 SD 書き込み失敗

目的:
- 書き込み途中失敗時の復帰確認

【推定】観点:
- 部分ファイルが残っても、
  次回保存を妨げないこと
- error 記録が残ること

### T23 readback 失敗

目的:
- LCD 読み出し失敗時の復帰確認

期待結果:
- 保存処理へ進まない
- game が再開する

### T24 busy 中再要求

目的:
- 多重要求抑止確認

手順:
1. 保存中に hotkey を連打する

期待結果:
- 追加要求は受理されない
- 同時に複数 capture しない
- ハングしない

### T26 `F5` reset 退避確認

目的:
- キー割り当て変更の退行確認

手順:
1. game 実行中に `F5` を押す
2. screenshot 動作を確認する
3. reset が発生しないことを確認する

期待結果:
- `F5`
  で screenshot が起動する
- `F5`
  で reset しない

### T27 `F1` reset 維持確認

目的:
- reset の残置確認

手順:
1. game 実行中に `F1` を押す

期待結果:
- `F1`
  で従来どおり reset が動く

### T25 DMA wait 異常

【推定】目的:
- timeout 導入時のフェイルセーフ確認

期待結果:
- timeout 後に処理が戻る
- busy が解除される

## 性能確認

計測項目:

- BMP 書き込み時間

評価:
- 絶対値の合否より、
  実プレイで許容できる停止時間かを確認する

### T32 read 時クロック条件

目的:
- LCD readback の安定動作条件確認

観点:
- read 時だけクロック低下が必要か
- dummy read 数が安定しているか

### T33 メモリ使用量

【推定】目的:
- 全画面バッファ方針が成立するか確認する

観点:
- capture バッファ確保で動作不安定にならないこと

## ログと観測ポイント

初期版では、
少なくとも次のログキーを固定して持つ。

ログ出力先と有効化方法は次で固定する。

- 出力先は既存の runtime log 経路とする
- screenshot 用ログは default off の debug log として扱う
- テスト時は screenshot debug log を有効化した build または設定で観測する
- release 向け default では常時表示しない

- `SS_REQ`
  : `F5`
    受付
- `SS_VBLANK`
  : VBLANK 起点で screenshot 要求を検出
- `SS_DMA_WAIT_BEGIN`
  : `lcd_dma_wait()`
    開始
- `SS_DMA_WAIT_DONE`
  : `lcd_dma_wait()`
    完了
- `SS_CAPTURE_BEGIN`
  : LCD readback 開始
- `SS_CAPTURE_DONE`
  : LCD readback 完了
- `SS_SAVE_BEGIN`
  : BMP 保存開始
- `SS_SAVE_DONE`
  : BMP 保存成功完了
- `SS_ERROR`
  : error code 記録

観測点は次で固定する。

- VBLANK 起点確認:
  `SS_REQ`
  の直後ではなく、
  次の
  `SS_VBLANK`
  以降に capture が始まること
- DMA 待ち確認:
  `SS_DMA_WAIT_BEGIN`
  と
  `SS_DMA_WAIT_DONE`
  が
  `SS_CAPTURE_BEGIN`
  より前に出ること
- readback 確認:
  `SS_CAPTURE_BEGIN`
  と
  `SS_CAPTURE_DONE`
  が対で出ること
- 保存確認:
  `SS_SAVE_BEGIN`
  と
  `SS_SAVE_DONE`
  が対で出ること
- 異常系確認:
  失敗時は
  `SS_ERROR`
  が出て、
  その後 game が復帰すること

## 完了条件

初期版では次を完了条件とする。

- 正常系 T01 から T04 が通る
- 整合性確認 T11 から T14 で重大破綻がない
- 異常系 T21 から T24 でハングしない
- 一時停止時間が実用上許容できる範囲である
